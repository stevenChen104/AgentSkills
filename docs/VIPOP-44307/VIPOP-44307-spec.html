Scope：patch

1. 背景

在 2026/03/19 的 demo 會議中，主管們回饋 ATS（Applicant Tracking System）在廣告暫停（Pause）或關閉（Off）狀態下的操作動線與 VIP 端差異過大，導致使用者誤以為系統發生異常。

現行大原則：廣告 Pause/Off 前已進入 ATS 甄試流程的履歷，仍可持續查看並進行流程管理，但 BC 溝通規則與 VIP 相同，不可發送。然而舊有介面並未明確告知用戶此限制，且部分按鈕直接隱藏，造成用戶困惑。

2026/04/27 版本調整：主要改為「顯示但 disable + 提供 CTA 引導開啟廣告」，使 ATS 與 VIP 動線一致，並加上後端阻擋防止漏洞。

2. 目標

  • 企業用戶能在廣告暫停/關閉時，清楚知道哪些功能受到限制及原因
  • 企業用戶能透過 CTA 快速前往開啟廣告，減少操作斷點
  • 避免使用者誤以為系統發生問題而產生客訴

3. User Stories

  • 身為企業 HR，我想要在廣告暫停/關閉時仍能看到邀約相關按鈕，以便我了解為何無法操作並知道如何解決
  • 身為企業 HR，我想要在新增行程的步驟中，提前知道無法通知求職者，以便我能在操作前決定是否先開啟廣告
  • 身為企業 HR，我想要在修改/刪除行程時收到清楚的警示，以便我確認對方已知行程異動
  • 身為企業 HR，我想要在取消錄用時看到明確提示，以便我確保求職者已知此異動

4. Use Cases

UC-1: 廣告暫停/關閉時嘗試使用邀約功能（詢問意願/邀約測評）

主要角色企業 HR（廣告處於暫停或關閉狀態）
前置條件企業廣告為暫停中或關閉中；履歷已進入甄試流程
後置條件（成功）使用者點擊「開啟廣告」CTA，另開分頁前往對應頁面
後置條件（失敗）使用者放棄操作，不進行任何流程推進

主要流程

  1. 用戶在看板/清單側邊卡片或履歷詳細頁點擊「邀約」按鈕
  2. 系統展開邀約選單，詢問意願、邀約測評顯示為 disabled 狀態
  3. 選單上方顯示提示文案：「廣告尚未開啟，開啟後即可使用完整功能」
  4. 使用者點擊 [開啟廣告] 按鈕
  5. 系統另開分頁導至對應頁面（依廣告狀態而定）

替代流程

  • 4a. 廣告為「暫停中」→ 另開分頁至 /rms/ad/status?isShowReOpenModal=1，直接顯示「立即恢復刊登」圖層
  • 4b. 廣告為「關閉中」→ 另開分頁至 /rms/order/general/selectPlan

特殊情境

若履歷卡片在「錄用階段」，邀約按鈕點擊後展開的選單中，詢問意願為 disabled，不顯示邀約測評

流程圖

flowchart TD
    A[用戶點擊邀約按鈕] --> B[展開邀約選單]
    B --> C{履歷是否在錄用階段?}
    C -->|否| D[顯示詢問意願/邀約測評, 均為 disabled]
    C -->|是| E[顯示詢問意願 disabled, 不顯示邀約測評]
    D --> F[上方顯示提示文案 + 開啟廣告按鈕]
    E --> F
    F --> G{用戶點擊開啟廣告}
    G -->|廣告暫停中| H[另開分頁至 /rms/ad/status?isShowReOpenModal=1\n直接顯示立即恢復刊登圖層]
    G -->|廣告關閉中| I[另開分頁至 /rms/order/general/selectPlan]

UC-2: 廣告暫停/關閉時新增行程（邀約面試/到職日）

主要角色企業 HR
前置條件廣告為暫停中或關閉中；進行卡片拖曳、移動至、或行事曆新增行程
後置條件（成功）行程成功新增，不發送 BC 溝通；使用者清楚知道求職者未收到通知
後置條件（失敗）使用者中途放棄或選擇先開啟廣告

主要流程

  1. 用戶拖曳卡片或選擇「移動至」進入邀約流程 Step 1
  2. Step 1 正常填寫行程資訊，按鈕為「新增」
  3. 進入 Step 2「通知求職者」
  4. 系統顯示：
    • 上方提示文字：「廣告未開啟，不提供聯絡求職者」
    • 通知求職者預設為「不通知求職者」且不可修改
    • 下方提示文案：「廣告尚未開啟，開啟後即可通知求職者」
    • [開啟廣告] 按鈕
  5. 用戶確認後進入 Step 3
  6. Step 3 顯示通知求職者圖示 + 提示文字：「廣告未開啟，不提供聯絡求職者」
  7. 完成，系統新增行程但不發 BC 溝通

流程圖

flowchart TD
    A[觸發新增行程\n卡片拖曳/移動至/行事曆] --> B[Step 1: 填寫行程資訊]
    B --> C[Step 2: 通知求職者]
    C --> D[顯示: 廣告未開啟，不提供聯絡求職者\n通知選項 locked = 不通知\n下方提示 + 開啟廣告按鈕]
    D --> E{用戶選擇}
    E -->|繼續| F[Step 3: 完成確認\n顯示不通知求職者圖示]
    E -->|點擊開啟廣告| G{廣告狀態}
    G -->|暫停中| H[另開分頁\n/rms/ad/status?isShowReOpenModal=1]
    G -->|關閉中| I[另開分頁\n/rms/order/general/selectPlan]
    F --> J[完成，行程新增，不發 BC 溝通]

UC-3: 廣告暫停/關閉時修改行程

主要角色企業 HR
前置條件廣告為暫停中或關閉中；行程尚未過期
後置條件（成功）行程成功修改，不發 BC 溝通；完成後顯示彈窗說明
後置條件（失敗）用戶放棄修改或先去開啟廣告

主要流程

  1. 用戶點擊修改行程
  2. 系統在修改畫面新增提示文案：「由於廣告未開啟，本次異動將不會通知求職者」
  3. 頁面同時顯示 [開啟廣告] 按鈕
  4. 用戶完成修改後送出
  5. 系統彈出完成彈窗（非僅 toast）：
    • 標題：「修改完成，但未通知求職者」
    • 內文：「由於廣告未開啟，本次異動不會通知求職者，請確保對方已知行程異動。」
  6. 行程修改完成，不發 BC 溝通

流程圖

flowchart TD
    A[用戶點擊修改行程] --> B[進入修改頁面]
    B --> C[顯示提示: 由於廣告未開啟，本次異動將不會通知求職者\n+ 開啟廣告按鈕]
    C --> D{用戶行為}
    D -->|繼續修改| E[填寫修改內容並送出]
    D -->|點擊開啟廣告| F{廣告狀態}
    F -->|暫停中| G[另開分頁\n/rms/ad/status?isShowReOpenModal=1]
    F -->|關閉中| H[另開分頁\n/rms/order/general/selectPlan]
    E --> I[完成彈窗\n標題: 修改完成，但未通知求職者\n內文: 說明不會通知求職者]
    I --> J[結束，不發 BC 溝通]

UC-4: 廣告暫停/關閉時刪除未過期行程 / 退回上一階段

主要角色企業 HR
前置條件廣告為暫停中或關閉中；行程尚未過期
後置條件（成功）行程刪除或退回，不發 BC 溝通；用戶已看到醒目警示
後置條件（失敗）用戶取消操作

主要流程

  1. 用戶點擊刪除未過期行程，或拖曳卡片/選擇「移動至」退回上一階段
  2. 系統彈出填寫原因的 pop-up
  3. Pop-up 中新增醒目提示文案（加上醒目底色）：「由於廣告未開啟，本次異動不會通知求職者，請確保對方已知行程取消。」
  4. 用戶確認操作
  5. 系統執行刪除/退回，不發 BC 溝通

流程圖

flowchart TD
    A[用戶點擊刪除行程\n或退回上一階段] --> B[彈出填寫原因 Pop-up]
    B --> C[顯示醒目提示文案\n廣告未開啟，不會通知求職者\n文字加上醒目底色]
    C --> D{用戶決定}
    D -->|確認| E[執行刪除/退回\n不發 BC 溝通]
    D -->|取消| F[關閉 Pop-up，不執行操作]

UC-5: 廣告暫停/關閉時取消錄用

主要角色企業 HR
前置條件廣告為暫停中或關閉中；到職日尚未過期
後置條件（成功）取消錄用成功，不發 BC 溝通；用戶已看到醒目警示
後置條件（失敗）用戶取消操作

主要流程

  1. 用戶在看板或錄用狀態頁（/rms/ats/hiring/status?jobNo={}）操作取消錄用
  2. 系統判斷到職日是否過期
  3. 若未過期：彈出確認 pop-up，顯示醒目提示文案（醒目底色）
  4. 用戶確認取消錄用
  5. 系統完成取消錄用，不發 BC 溝通

流程圖

flowchart TD
    A[用戶選擇取消錄用\n看板或錄用狀態頁] --> B[判斷到職日是否過期]
    B -->|已過期| C[直接完成取消錄用，不發 BC 溝通]
    B -->|未過期| D[彈出確認 Pop-up\n顯示醒目提示文案]
    D --> E{用戶決定}
    E -->|確認| F[完成取消錄用\n不發 BC 溝通]
    E -->|取消| G[關閉 Pop-up，維持錄用狀態]

10. 驗收標準

UC-1 邀約按鈕相關

  • 廣告暫停/關閉時，點擊邀約按鈕，詢問意願和邀約測評顯示為 disabled

  • 廣告暫停/關閉時，修改行程頁面顯示「由於廣告未開啟，本次異動將不會通知求職者」

  • 後端確認取消錄用完成但未發出 BC 溝通
